\section{Motivation and Requirements}

\subsection{Behavioral Contracts in the Wild}

We collected \texttt{CLAUDE.md} and \texttt{AGENTS.md} files from 50~popular
open-source projects and extracted candidate behavioral rules. These files
contain many imperative statements, but only a subset are OS-observable
behavioral contracts. A rule such as ``never use \texttt{any}'' is a
coding-style directive and is outside ActPlane's scope. A rule such as ``never
push directly to \texttt{main}'' or ``run tests before committing'' has an
observable process and syscall footprint and can be enforced as a harness
contract.

The corpus serves two purposes. First, it grounds the motivation: developers
already write behavioral contracts for agents. Second, it derives evaluation
workloads: we convert rules that are observable, expressible, robust, and
temptable into policy/task pairs (\S\ref{sec:evaluation}), reporting both the
full corpus and the filtered evaluation subset.

\subsection{Why Tool-Layer Guards Are Insufficient}

Agent frameworks and permission systems can intercept tool calls and apply
deterministic policies~\cite{wang2025agentspec,shi2025progent}. This is useful,
but it assumes that effects pass through the guarded tool boundary. A coding
agent violates that assumption whenever it invokes a shell, starts a subprocess,
loads a library, or talks to an external service from generated code. The same
semantic action, for example invoking \texttt{git push --force}, can appear as a
direct shell command, a nested \texttt{bash -c} string, a Python subprocess, or
a helper binary. A tool-layer guard sees only the outer tool invocation.

The OS sees the effect itself. Every exec, file open, unlink, rename, truncate,
and socket connect crosses a kernel boundary. If a policy is phrased over those
effects and over the lineage of the agent process tree, then the policy applies
regardless of the high-level route the agent selected.

\subsection{Requirements}

ActPlane is designed around five requirements.

\textbf{R1: Whole-execution scope.} A rule applies to the agent's
descendants, not just to the top-level process or the initial tool call.

\textbf{R2: Cross-object provenance.} Many contracts involve information flow,
not a single event. Reading a secret file taints the reader; writing a
derived file taints that file; reading the derived file later taints
the next process.

\textbf{R3: Effect semantics that match the backend.} \texttt{block} means
pre-operation denial and is implemented only by BPF-LSM hooks. Tracepoints
support \texttt{audit} and \texttt{kill} but not \texttt{block}.

\textbf{R4: Corrective feedback.} The harness not only fails a violating
action but tells the agent which rule fired, why, and what alternative or
precondition allows progress.

\textbf{R5: Deployable mechanism.} The implementation uses upstream Linux
interfaces: eBPF for dynamic loading and BPF-LSM for pre-operation deny, rather
than a custom kernel module.
