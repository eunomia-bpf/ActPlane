## Problem

Indirect prompt injection (PIA): adversary embeds malicious instructions in tool
outputs (e.g., a malicious email body). The agent's LLM cannot reliably distinguish
data from instructions, so it may exfiltrate data or trigger unauthorized actions.
Existing defenses (model alignment, input/output filters, human-in-the-loop) are
probabilistic and provide no deterministic guarantee.

## Core idea: IFC at the planner layer

Attach **confidentiality** and **integrity** labels to every piece of data the agent
touches. Labels form a lattice; the join (⊔) computes the least upper bound. Two
orthogonal dimensions:

- **Integrity**: L = {T, U} with T ⊑ U. T = trusted (user query, system prompt),
  U = untrusted (tool results from external sources). Prevents untrusted data from
  influencing control-flow decisions (consequential actions).
- **Confidentiality**: L = readers lattice P(U). A label is the set of authorized
  readers. Join = set intersection (fewer readers = more restrictive). Prevents data
  from flowing to unauthorized recipients.

Product lattice: (integrity, confidentiality). Top = (U, H) = untrusted + secret.
Bottom = (T, L) = trusted + public.

## Label propagation

Labels are attached to:
- **Messages** in conversation history (system/user = ⊥, tool results = labeled by tool wrapper)
- **Variables** in planner memory (each variable x has label τ(x))
- **Tool call arguments** (each arg carries its own label)

Propagation rule (Algorithm 5, line 9): when tool f is called, the result label is:
```
ℓ'' = ⊔{τ(x) : x ∈ R(f)} ⊔ ℓ_f ⊔ ⊔{ℓ'_a : a ∈ args}
```
i.e., join of all datastore variables the tool reads, the tool's own label, and all
argument labels. All variables the tool may write get this label.

## Two fundamental policies

1. **P-T (Trusted Action)**: a consequential tool call (send_message, set_event, etc.)
   may proceed only if the tool label ℓ_f = (T, ⊤), meaning the decision to call it
   was made in a context containing only trusted data. Formally: ℓ_f ⊑ π_f and
   ∀x ∈ args: ℓ'_x ⊑ π_x.

2. **P-F (Permitted Flow)**: an egress tool call (send data to recipients R) may proceed
   only if all recipients are in the data's authorized reader set. Expressed as
   π_d = (⊤, R). Prevents illicit data flows regardless of trust context.

Enforcing both P-T and P-F gives:
- **Non-interference** for integrity (untrusted data cannot trigger consequential actions)
- **Explicit secrecy** for confidentiality (no direct unauthorized data flows; implicit
  flows via control-flow ordering are allowed as a pragmatic trade-off)

## FIDES: advanced IFC mechanisms

The basic planner with taint-tracking has a limitation: once untrusted data enters the
conversation history, the entire context is tainted, blocking all subsequent consequential
actions. FIDES solves this with two novel primitives:

### 1. Selective variable introduction (HIDE)

When a tool returns untrusted data, instead of appending it to the conversation:
- Check if any node in the result has a label more restrictive than current context
- If so, store it in a fresh variable in planner memory with its original label
- The conversation history label stays clean (trusted)
- The planner can reference the variable by name in future tool calls without tainting its context

Example: `read_emails()` returns untrusted content → stored as `#read_emails_0` → planner
can call `send_message(to=user, message=#read_emails_0)` without the send_message decision
being tainted.

### 2. Constrained inspection (query_llm + inspect)

When the planner needs to examine untrusted data to make a decision:
- `query_llm(prompt, input=#var, output="bool")` — sends the variable to a **quarantined
  LLM** (isolated, no tool access) with a constrained output schema
- The quarantined LLM returns a typed result (bool, enum, structured dict)
- `inspect(#query_llm_0)` — reveals the constrained result to the planner

Key insight: **type information as declassification**. A bool extracted from untrusted
data carries label ((U, L), bool). Since a bool can only carry 1 bit, it is much less
useful for prompt injection payloads or data exfiltration. The type lattice (bool ⊑
enum ⊑ string) integrates into the security lattice, allowing policies to accept
low-capacity untrusted values in consequential actions.

## Task taxonomy

| Category | Description | Example |
|----------|-------------|---------|
| **DI** (Data Independent) | Tool call sequence doesn't depend on tool results | "Read top 3 emails, send to user" |
| **DIQ** (DI with query_llm) | Like DI but needs query_llm to extract structured data | "Summarize emails, send summary" |
| **DD** (Data Dependent) | Tool sequence depends on data content | "If there's a meeting request, create calendar event" |

- Basic planner: can do DI tasks but fails P-T on most (context tainted after read)
- Variable Passing: can do DI safely (pass by reference), fails DIQ/DD
- FIDES: can do DI + DIQ safely; DD possible when plan doesn't violate policy

## Evaluation on AgentDojo

**Security (Attack Success Rate)**:
- Basic planner (no policy): 156 successful injections (gpt-4o)
- FIDES (no policy): 23 injections (taint-tracking alone blocks most)
- Basic* (with P-T/P-F): 0 injections
- FIDES* (with P-T/P-F): 0 injections (1 edge case)

**Utility (Task Completion Rate)** with policy checks enabled:
- FIDES outperforms Basic planner for all models
- With reasoning models (o1, o3, o4-mini): up to 16.7% better than Basic
- With prompt tuning: up to 24% better, approaching human oracle performance
- Without policy: FIDES ≈ Basic planner (no utility cost from the extra mechanisms)
- FIDES performs 57.52% better than Variable Passing planner (o1)

**Key findings**:
1. With policy checks, FIDES blocks all attacks (0 ASR) while maintaining high utility
2. Without policy, taint-tracking alone reduces attacks from 156 to 23
3. FIDES's hiding + inspection primitives expand the class of securely completable tasks
4. Reasoning models are significantly better at using query_llm correctly

## Comparison with ActPlane

FIDES is the **most directly comparable** system to ActPlane — both apply labeled IFC
to agent security. The comparison reveals complementary strengths:

| Dimension | FIDES | ActPlane |
|-----------|-------|----------|
| **Enforcement layer** | Application (planner/planning-loop) | OS kernel (eBPF/BPF-LSM) |
| **Label model** | Lattice: (integrity, confidentiality) × type | Flat u64 bitmask with boolean algebra (req/forbid) |
| **Label granularity** | Per-value, per-field in tool results | Per-process, per-file, per-endpoint |
| **Propagation** | Through conversation history + variable memory | Through fork/exec/read/write/connect syscalls |
| **What flows** | Data in tool call arguments and results | OS-level operations (file I/O, network, exec) |
| **Declassification** | Type-based: constrained output schema limits information capacity (bool < enum < string) | Gate-based: `declassify L by exec "program"` |
| **Temporal constraints** | None | `after G since S` with staleness epochs |
| **Security guarantees** | Formal: non-interference (integrity), explicit secrecy (confidentiality) | Empirical: 12 worked examples |
| **Bypass resistance** | Can be bypassed by direct syscalls, subprocesses, tool-layer escapes | Cannot — enforces below tool layer at syscall boundary |
| **Threat model** | Indirect prompt injection via tool results | Arbitrary agent misbehavior including tool/subprocess escape |
| **Policy specification** | Per-tool labels (P-T, P-F) assigned in tool wrappers | Custom DSL with source/rule/declassify/gate |
| **Hiding/revealing** | Novel: selective variable introduction + quarantined LLM inspection | Not applicable (kernel doesn't see LLM context) |
| **Evaluation** | AgentDojo (97 user tasks × 35 injection tasks × 5 models) | 12 worked examples + e2e tests |

### Key insights for ActPlane

1. **FIDES's type-based declassification** is elegant — extracting a bool from untrusted
   data limits exfiltration capacity to 1 bit. ActPlane's `declassify` is gate-based
   (exec a specific program) which is coarser but enforced at kernel level.

2. **FIDES's formal guarantees** (non-interference, explicit secrecy) are stronger than
   ActPlane's empirical validation. ActPlane could benefit from similar formal analysis.

3. **Complementary layers**: FIDES handles intra-LLM data flow (what the planner sees
   and decides); ActPlane handles OS-level flow (what the process actually does). An
   agent could use FIDES at the planner layer AND ActPlane at the OS layer — defense
   in depth. FIDES catches prompt injection at the planning stage; ActPlane catches
   anything that escapes the planner (direct syscalls, subprocess spawning, tool-layer bugs).

4. **FIDES cannot enforce below the tool layer**: if an agent spawns a subprocess, writes
   to a file, or makes a raw network connection outside the tool API, FIDES has no
   visibility. ActPlane's eBPF hooks catch all of these.

5. **FIDES's variable-passing planner** is analogous to ActPlane's label propagation:
   both track provenance through intermediaries. But at different granularities — FIDES
   tracks per JSON field; ActPlane tracks per process/file/endpoint.

6. **Implicit flows**: FIDES explicitly allows them (explicit secrecy, not full
   non-interference for confidentiality) as a pragmatic trade-off. ActPlane also cannot
   prevent implicit flows (the kernel sees operations, not branch decisions). The
   Causality Laundering paper (2604.04035) identifies this as an exploitable gap.
