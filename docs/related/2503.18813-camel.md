# CaMeL: Defeating Prompt Injections by Design

- **Authors**: Edoardo Debenedetti, Ilia Shumailov, Tianqi Fan, Jamie Hayes, Nicholas Carlini, Daniel Fabian, Christoph Kern, Chongyang Shi, Andreas Terzis, Florian Tramer
- **Affiliation**: Google, Google DeepMind, ETH Zurich

- **PDF**: [2503.18813.pdf](2503.18813.pdf)
- **Code**: https://github.com/google-research/camel-prompt-injection

## Summary

CaMeL creates a protective system layer around the LLM that separates trusted control
flow from untrusted data. Inspired by Control Flow Integrity, Access Control, and
Information Flow Control. Associates *capabilities* (metadata) to every value to
restrict data and control flows via fine-grained security policies.

## Key design

- Extracts control and data flows from user queries
- Untrusted data retrieved by the LLM can never impact program flow
- Custom Python interpreter enforces security policies at tool-call boundaries
- Does not rely on model behavior modification
- 77% of tasks solved with provable security on AgentDojo (vs 84% undefended)

## Relevance to ActPlane

CaMeL's capability-based IFC is the closest application-layer analog to ActPlane's
kernel-level labeled IFC. Key differences:
- Enforcement layer: Python interpreter (CaMeL) vs eBPF kernel (ActPlane)
- Granularity: per-value capabilities (CaMeL) vs per-process/file/endpoint labels (ActPlane)
- Threat model: prompt injection (CaMeL) vs arbitrary tool/syscall escape (ActPlane)
- CaMeL requires a custom interpreter; ActPlane is transparent to the agent's toolchain
- Complementary: CaMeL handles intra-LLM data flow; ActPlane handles OS-level information flow
