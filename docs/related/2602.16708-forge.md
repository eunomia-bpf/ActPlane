# FORGE: Formal Policy Enforcement for Real-World Agentic Systems

- **Authors**: Nils Palumbo, Sarthak Choudhary (UW-Madison), Jihye Choi (UW-Madison), Guy Amir (Cornell), Prasad Chalasani (Langroid), Somesh Jha (UW-Madison)
- **Arxiv**: https://arxiv.org/abs/2602.16708
- **PDF**: [2602.16708.pdf](2602.16708.pdf)

## Summary

FORGE frames agent policy enforcement as a cross-cutting concern (aspect-oriented
programming). Policies are written in **Datalog** over abstract predicates describing
the execution context. An observability service governed by assume/guarantee contracts
maintains predicates, and a reference monitor consults the policy at each action.

## Key design

- Policy language: **Datalog** — declarative, supports recursion for transitive relationships, deterministic
- Aspect-oriented enforcement: policies independent of agent logic
- Static analyses: contradiction detection, redundancy, subsumption, conditional reachability
- Assume/guarantee contracts for the observability service
- Evaluated on: IFC for prompt injection defense, approval workflows, organizational policies

## Relevance to ActPlane

Both use declarative policy languages for agent enforcement. Key differences:
- Policy language: Datalog (FORGE) vs custom DSL (ActPlane)
- Enforcement layer: application-level reference monitor (FORGE) vs kernel eBPF (ActPlane)
- FORGE's Datalog supports recursive/transitive queries; ActPlane's DSL is non-recursive but has temporal gates
- FORGE has formal static analysis (contradiction/redundancy detection); ActPlane lacks this
- FORGE explicitly handles multi-agent scenarios; ActPlane is single-agent focused
- FORGE's case study includes IFC for prompt injection — directly comparable to ActPlane's label model
- Bypass resistance: FORGE enforces at tool-call API; ActPlane at kernel boundary
