# Agent-C: Enforcing Temporal Constraints for LLM Agents

- **Authors**: Adharsh Kamath, Sishen Zhang, Calvin Xu, Shubham Ugare, Gagandeep Singh, Sasa Misailovic (UIUC)
- **Arxiv**: https://arxiv.org/abs/2512.23738
- **PDF**: [2512.23738.pdf](2512.23738.pdf)
- **Code**: https://github.com/structuredllm/agent-c

## Summary

Agent-C introduces a domain-specific language for expressing temporal safety properties
(e.g., "authenticate before accessing data"), translates specifications to first-order
logic, and uses SMT solving to detect non-compliant agent actions during token
generation. Leverages constrained generation to ensure compliance.

## Key design

- DSL for temporal properties over agent action sequences
- Translation to first-order logic + SMT solving
- Constrained generation: intervenes during token generation to prevent non-compliant actions
- Generates compliant alternatives when a non-compliant action is attempted
- 100% safety (conformance) on customer service and airline ticket reservation benchmarks

## Relevance to ActPlane

Both address temporal ordering constraints on agent actions. Key differences:
- Enforcement layer: token generation (Agent-C) vs kernel syscall (ActPlane)
- Temporal model: first-order logic + SMT (Agent-C) vs epoch counters with staleness gates (ActPlane)
- Agent-C is more formally expressive for temporal properties; ActPlane's `after G since S` is simpler but kernel-efficient
- Agent-C intervenes at LLM output; ActPlane intervenes at OS execution
- Agent-C cannot enforce against direct syscalls or subprocess escapes
- Both have DSLs for policy specification
